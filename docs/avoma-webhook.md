# Avoma SNS webhook

## Avoma SNS webhook + opportunity-analysis agent (added 2026-05-28)
AWS pipeline `Avoma → Lambda → DynamoDB → SNS → Replit /webhook`. Receiver side only — publisher lives AWS-side.

- **`POST /webhook`** handles SubscriptionConfirmation (auto-GETs SubscribeURL), Notification, UnsubscribeConfirmation. SNS `Message` is JSON-encoded; `meeting_id` lives at top level (also fallback to `uuid` / `full_data.uuid`). Returns 200 fast (<1s) so SNS doesn't retry. **Hardened 2026-05-29** (helpers + gates at `server.py` ~line 4489): every message passes fail-closed gates **before any side effect** (disk write / SubscribeURL GET / enrichment): (1) **TopicArn allowlist** `_SNS_ALLOWED_TOPIC_ARNS` (env `SNS_ALLOWED_TOPIC_ARNS`, comma-sep; default the dev topic `arn:aws:sns:us-east-1:385817851343:avoma-meeting-events-dev` — **set this env to add the prod topic or the webhook 403s it**); (1b) **region + account cross-check** (defense-in-depth, added 2026-05-29) — `_sns_arn_region_account` parses the TopicArn and rejects unless region ∈ `_SNS_ALLOWED_REGIONS` (env `SNS_ALLOWED_REGIONS`, default `us-east-1`) and account ∈ `_SNS_ALLOWED_ACCOUNT_IDS` (env `SNS_ALLOWED_ACCOUNT_IDS`, default `385817851343`); these are belt-and-suspenders behind the exact-ARN match and matter when the topic allowlist is later broadened. (2) **freshness** `_sns_is_message_fresh` (env `SNS_MAX_MESSAGE_AGE_MINUTES`, default 15, anti-replay, ±5min clock skew); (3) **AWS SNS signature verification** `_sns_verify_signature` (SignatureVersion strictly 1=SHA1 / 2=SHA256 — unknown versions rejected; RSA PKCS1v15; fetches the X.509 cert from `SigningCertURL` — **host-pinned, cached per-URL** — via `load_pem_x509_certificate(...).public_key()`; toggle `SNS_VERIFY_SIGNATURE=false` to disable, NOT recommended since TopicArn is spoofable in the body). On any gate failure → `403`. **Cert/SubscribeURL host pinning tightened 2026-05-29**: `_sns_host_allowed` now requires host to match `^sns\.<region>\.amazonaws\.com$` AND `<region> ∈ _SNS_ALLOWED_REGIONS` (was the over-broad `*.amazonaws.com`, which permitted a forged-cert bypass on any AWS-owned host) — applied to both the `SigningCertURL` cert fetch and the `SubscribeURL` confirmation GET → closes SSRF (e.g. `169.254.169.254`) and cross-region cert fetches. Verified 2026-05-29: unknown-topic/wrong-region/wrong-account/stale/bad-sig/non-SNS-cert-host/cross-region-cert/bad-SignatureVersion all → 403; crypto canonical-string + tamper-detection + strict host regex unit-tested.
- **Lean enrichment pipeline (rewritten 2026-05-28):** per Notification, `_enrich_bg`:
  1. Calls `sf_meeting_enricher.extract_sf_ids_from_meeting(meeting_uuid)` — one Avoma GET that returns `{meeting, opportunity_id, account_id, contact_ids[]}` from `crm_associations` (list of `{crm_obj_type, crm_obj_id}` typed dicts — `"oppo"`=Opp, `"account"`=Account, `"contact"`=Contact; **NOT** the keyed `{crm_opportunity_ids:[], ...}` shape).
  2. Persists meeting metadata + SF ids to `avoma_event_reports` with `status` of `no_sf_links` / `completed_no_opportunity` / `pending_analysis`.
  3. If `opportunity_id` resolved, hands off to the **opportunity-analysis agent** (`opportunity_analyzer.py`) — a `create_deep_agent` loaded with `prompts/opportunity_analysis_system_prompt.md` and scoped to ONLY the Salesforce + Avoma MCP tools (27 total, pulled from `agent_manager._cached_mcp_tools_by_server`). Defaults to `anthropic:claude-sonnet-4-5` (override via `OPP_ANALYZER_MODEL`), 16000 max_tokens (`OPP_ANALYZER_MAX_TOKENS`), outer `asyncio.wait_for` (`OPP_ANALYZER_TIMEOUT_S`, default **900s** — raised from 600 on 2026-05-29; the env override + `OPP_ANALYZER_RECURSION_LIMIT` are now actually wired, previously hardcoded). Returns one JSON record matching `OpportunityAnalysisRecord` (meta + conflicts + win_likelihood + evidence + ...). Robust JSON extraction: strict → fence-strip (handles missing closing fence) → largest balanced `{...}` window; on total failure persists `{_error, _raw, _raw_len}` for forensics.
  - **Failing-report fixes (2026-05-29):** reports were showing bare `status="failed"` with no reason. Four fixes: **(1) error detail persisted** — both the webhook `_enrich_bg` and the `/reanalyze` `_bg` write paths now store `opportunity_analysis_status = f"{status}: {error}"` when status≠`completed` (was bare `an["status"]`, discarding `an["error"]`), so the UI shows e.g. `failed: timeout after 900s` instead of `failed`. **(2) summariser skipped during analysis** — new `_skip_llm_summarizer` ContextVar (server.py); `analyze_opportunity` sets it True around `agent.ainvoke` (reset in `finally`), and `_wrap_mcp_tool`'s two summariser call sites bypass the gpt-4o-mini summariser (which timed out at 45s/call and fell back to truncation anyway — pure waste) for deterministic truncation. Main chat agent unaffected (default False). **(3) context-trim middleware on the analyzer** — `_get_agent` now passes `ContextTrimMiddleware` (same one the main agent uses, gated on `CONTEXT_TRIM_ENABLED`) instead of `middleware=[]`; without it the analyzer replayed every accumulated tool_result each LLM step until a single Anthropic call exceeded `LLM_REQUEST_TIMEOUT_S` and died with `APITimeoutError`. **(4) Avoma 429 backoff** — `avoma_mcp_server._send_with_retry` (shared by `_get`/`_patch`) and `sf_meeting_enricher.extract_sf_ids_from_meeting` now retry on HTTP 429, honouring `Retry-After` (else exponential backoff, capped 30s, up to `AVOMA_MAX_429_RETRIES`=5); the backoff sleep happens outside `_api_lock`. Avoma rate-limiting during `extract_sf_ids_from_meeting` was the direct cause of several `status=failed` reports. NOTE: very large opps (many meetings × transcript/notes/insights reads) can still exceed 900s of sequential agent work and time out cleanly; further speedups (scope reduction / faster model / async fan-out) are a separate perf task.
- **Removed 2026-05-28:** the legacy 3-tier SOQL chain (`deal_health` / `account_briefing` / `full_snapshot`) + per-tier `gpt-4o-mini` summaries + `cache_sync.update_cache_from_report` hook. The corresponding columns on `avoma_event_reports` (`deal_health_data`, `account_briefing_data`, `full_snapshot_data`, `*_summary`) are no longer written by the webhook (kept in schema for backfill access). `pull_deal_health` / `pull_account_briefing` / `pull_full_snapshot` / `summarize` / `enrich_and_summarize` in `sf_meeting_enricher.py` are now dead code reachable only if other callers import them. `cache_sync` still feeds the 3 cache tables via `POST /api/cache/bulk_import*` and `GET /cron/sync-sf-to-cache`.
- **Supabase table `public.avoma_event_reports`** (indexes on `created_at desc`, `sf_opportunity_id`, `sf_account_id`, `(status, created_at desc)`): `message_id` UNIQUE for SNS dedup, raw SNS envelope in `raw_sns_envelope jsonb`. New analyzer columns (added 2026-05-28): `opportunity_analysis_data jsonb`, `opportunity_analysis_status text` (`running` | `completed` | `parse_error` | `failed: ...`).
- **Endpoints**: `GET /api/avoma/reports?limit=&status=` (lightweight list), `GET /api/avoma/reports/{message_id}` (full row incl. analyzer record), `POST /api/avoma/reports/{message_id}/reanalyze` (fire-and-forget rerun of the analyzer on an existing row; returns `{status:"started", poll_url}` immediately so curl client disconnect doesn't kill the task).
- **Agent-side access:** `custom_tools/avoma_reports.py` `get_avoma_report_detail` includes the analyzer columns in its select so the main chat agent can read the JSON record from any subsequent turn.
