# Cache layer & Opportunity Observatory

## Cache layer (added 2026-05-28) — `cache_sync.py`
Pre-processed 50-100ms query layer over 3 new Supabase tables, hooked into the same enrichment pipeline. Bulk-seeded from `attached_assets/deal_summaries_bulk_parallel_*.json` (1,104 opps).
- **`public.opportunity_cache`** (PK `opportunity_id`): current state (amount, stage, close_date, probability, owner) + pre-calculated `health_score` / `momentum` (Active≥75 | Moderate 50-74 | Slow 25-49 | Stalled<25) / `days_in_stage` / `risk_signals jsonb`. GIN tsvector on `(opportunity_name || account_name)`, BTREE on stage/amount/health/momentum. `data_source` distinguishes `bulk_backfill` / `webhook_update` / `sf_poll`.
- **`public.meeting_cache`** (PK `meeting_uuid`): 1 row per meeting; `opportunity_ids[]` + summary + report ref.
- **`public.field_history_cache`** (UNIQUE `(opp,field,changed_date,old,new)`): every `OpportunityFieldHistory` + `stage_history` row, deduped via `upsert(ignore_duplicates=True)`. Captures ALL field changes — same storage cost, big query value.
- **Health formula:** `stage_progress = Probability*0.5` + `recency = 30/(days_since_last_meeting+1)` + `activity = min(open_tasks,5)*4` ± `momentum_delta` (±10 for stage advance/regress last 30d), clamped 0-100.
- **Sync paths:**
  1. `cache_sync.update_cache_from_report()` called at the tail of `_enrich_bg` in server.py — after every webhook enrichment writes to `avoma_event_reports`, the 3 caches are updated. Logs `[CACHE-SYNC]`. Never raises.
  2. `GET /cron/sync-sf-to-cache?lookback_minutes=20` — LIGHT poll of `OpportunityFieldHistory` for recent SF-side changes (no Avoma meeting needed); refreshes only affected opp_cache rows. Synchronous. Wire to external scheduler if desired.
  3. `GET /cron/sf-pull-refresh?lookback_hours=72&limit=25&opp_only=true` (added 2026-05-29) — HEAVY companion that re-runs the full deterministic 3-tier pull (`_run_sf_pull_and_cache` → deal_health/account_briefing/full_snapshot + all 3 caches) for recently-seen opps, keeping the richer cache fresh between meetings (Opportunity CDC is off). Scans `avoma_event_reports` by `created_at` within the window, dedupes to the most recent report per `sf_opportunity_id`, then runs pulls **sequentially in a background task** (avoids parallel SOQL hammering) and returns `{status, queued, opportunities[]}` immediately — poll each report's `sf_pull_status`. `opp_only=false` also includes account/contact-only reports. Logs `[CRON-SF-PULL]`.
  4. **Nightly in-process scheduler (added 2026-05-29)** — `_nightly_sf_pull_scheduler()` (registered in `startup_event`). Because the deployment is **VM (always-running)**, an asyncio loop fires `_run_nightly_sf_pull()` once per day at the configured hour (default **00:00 UTC**), no separate Scheduled Deployment needed. Each run re-uses the same shared helpers as the endpoint (`_select_sf_pull_targets` → `_run_sf_pull_batch`) and honours the same `_SF_PULL_CRON_RUNNING` overlap guard. Scoped wide for "all relevant details": `SF_PULL_CRON_LOOKBACK_HOURS` (default **26**, covers a full day + buffer) × `SF_PULL_CRON_LIMIT` (default **500**), `opp_only=true`. Env overrides: `SF_PULL_CRON_ENABLED` (default true), `SF_PULL_CRON_HOUR` (0-23), `SF_PULL_CRON_TZ` (IANA, e.g. `America/New_York`; bad tz → UTC fallback). Logs `[NIGHTLY-SF-PULL] next run at … — sleeping Ns`, then `[NIGHTLY-SF-PULL]` batch lines. The endpoint (`/cron/sf-pull-refresh`) remains available for manual/ad-hoc triggers.
- **Endpoints:**
  - `POST /api/cache/bulk_import` — one-time opportunity backfill from JSON.
  - `POST /api/cache/bulk_import_field_history` — one-time backfill from `attached_assets/field_history_cache_*.json` (10,446 rows / 878 opps imported 2026-05-28). Idempotent via UNIQUE dedup.
  - `GET /api/cache/tenant/{opp_id}?meetings_limit=10` — full current state for one opp in one parallel-fanout call. `cache_sync.get_tenant_state` fires the 3 cache reads concurrently via `asyncio.gather` + thread executor (supabase-py is sync). Latency ≈ max of the 3 reads (~260ms) instead of sum (~785ms) — **3.0× faster than sequential, ~3.8× faster than live SF tier-1 SOQL chain, ~115× faster than full 3-tier `/webhook` enrichment**. Includes a `history_summary.by_field` rollup so callers don't re-scan.
  - `POST /api/ceo-query` — thin keyword router for `stalled` / `no meetings` / `amount history` / name search with stopword-stripping regex. For arbitrary questions, use the agent + `supabase_query` MCP tool against the cache tables.
  - **`POST /api/ask` + `GET /ask` UI (added 2026-05-29)** — natural-language Q&A answered ONLY from cached data + reports (`cache_qa.py`). A small gpt-4o-mini tool-calling loop (`CACHE_QA_MODEL`, `CACHE_QA_MAX_ITERS` default 6) with 7 read-only `StructuredTool`s **hard-scoped** to an allowlist of 5 tables (`opportunity_cache`, `meeting_cache`, `field_history_cache`, `avoma_event_reports`, `opportunity_observatory`) — table names are module constants, never taken from the model; `select(...)`-only, no write path, never touches live SF/Avoma. Sync supabase-py reads wrapped in `run_in_executor`. Returns `{answer, steps[], model}`; `GET /ask` is a self-contained dark-theme HTML page (text box + example chips + "N data lookup(s)" detail). **Unauthenticated, uses the service-role client — same posture as `/api/ceo-query` and `/avoma/reports`; gate behind auth before exposing publicly.**
- **`_parse_dt` always tz-aware** (assumes UTC if no tz info) — prevents the "can't compare offset-naive and offset-aware" error that bit on the first SABIC replay.
- Smoke-tested 2026-05-28: SABIC replay → opp_cache (health 44.3 / Slow / days_in_stage 21 / probability 20% / owner Karson Keogh), meeting_cache (1 row), field_history_cache (12 rows: 2 stage_history + 10 opp_field_history).
- Smoke-tested live: Sabic Opp `006P700000XjHg3IAF` meeting → 37s enrichment, 3 tiers populated, summaries cite real $ amount, real contacts, real activities. Internal standup with no opp linkage → completed in 22s, tier 2/3 produced account briefing on attendee's company, tier 1 correctly skipped.

## Opportunity Observatory (added 2026-05-29)
Pre-computed long-form opportunity dossiers, stored once and read by the agent through a **single-table-scoped** wrapper (the whole point: the agent can read these without the generic `supabase_query` MCP tool, so it cannot pivot to any other table).
- **`public.opportunity_observatory`** (PK `opportunity_id`): one row per opportunity. Header fields (`name`, `opportunity_owner`, `close_date` text, `amount` numeric, `stage`, `account_name`) + 8 long-form markdown sections: `sf_90day_evidence`, `avoma_evidence`, `outbound_campaign_intelligence`, `bundle_a_deal_progress`, `bundle_b_competition_fit`, `bundle_c_stakeholder_map`, `bundle_d_vulnerabilities`, `diagnosis_sheet`. Indexes: BTREE on `account_name` + `stage`, GIN tsvector on `(name || account_name)`. `created_at` / `updated_at` timestamptz.
- **Seeded** from `attached_assets/Observatory_for_Opportunities_-_Set_1_*.csv` (32 dossiers) via `scripts/import_observatory.py` — idempotent upsert on `opportunity_id`; DDL through the Management API, rows through the REST upsert (`Prefer: resolution=merge-duplicates`). `amount` parsed (strips `£`/`$`/commas; non-numeric → null). Re-run any time to refresh / add sets.
- **Agent access — `custom_tools/opportunity_observatory.py`** (3 read-only `@tool`s, all hard-scoped: `_TABLE = "opportunity_observatory"` is a module constant, never taken from the model):
  - `list_opportunity_dossiers(limit, stage, account_name_contains, name_contains)` — lightweight header rows only (omits the heavy markdown).
  - `get_opportunity_dossier(opportunity_id, sections=None)` — one full dossier; optional comma-separated subset of the 8 section names. **Section names validated against a fixed allowlist (`_VALID_SECTIONS`)** so the model can't inject arbitrary columns; unknown names → error.
  - `search_opportunity_dossiers(query, limit)` — fuzzy ILIKE over `name` + `account_name`.
  - All three GET only `…/rest/v1/opportunity_observatory`; no write path, no table parameter. Verified registered (273 total agent tools) + smoke-tested (list/search/get/section-guard) 2026-05-29.
