# Handoff — MASE changes (2026-06-18)

Catch-up doc for any agent/engineer picking up MASE. This captures the current state
after a large round of changes. Pair it with `CHANGELOG.md` (both repos) and `AGENTS.md`.
Repos: backend `mase_backend` (ECS, branch `gh-clean`), frontend `MASE` (Vercel, `main`).

---

## 1. Agent prompts live in SUPABASE (source of truth)

`public.jarvis_settings`, one row per agent (via `agent_prompt_store.py`):

| Agent | id key | Edit from |
| --- | --- | --- |
| Deal Intelligence Engine **sweep** (`deal_engine_sweep.py`) | `mase_deal_sweep` | Admin → **Deal Sweep** · `POST /api/deal-engine/sweep/prompt` |
| **Todo Runner** ("Run with AI" drafting; `components/agent/AgentRun.tsx`) | `mase_todo_runner` | Admin → **Todo Runner** · `POST /api/deal-engine/todo-runner/prompt` |
| Chat agent | `mase_chat_agent` | chat page panel · `POST /api/deal-engine/chat/prompt` |

- **Edit the Supabase prompt to change behaviour — NOT the `prompts/*.md` files.** Those
  are DEPRECATED cold-start seeds (carry a banner stripped at load by
  `agent_prompt_store.strip_leading_banner`).
- Sweep: `_load_prompt` = Supabase-first, disk-seed fallback; `_get_agent` re-resolves on
  a 15s TTL and rebuilds when the prompt fingerprint changes (API + worker pick it up).
- Todo-runner: `AgentRun.tsx` fetches the effective prompt per run; `DRAFTING_SYSTEM_PROMPT`
  is just the offline fallback.
- Verified the sweep agent uses the **Salesforce (12) + Avoma (15) MCP tools** and reads
  Avoma transcripts (verbatim, dated, speaker-attributed quotes in MEDDPICC).

## 2. MASE Knowledge = an ISOLATED system (NOT VIBE projects/documents)

- Own tables: `mase_documents` + `mase_document_chunks` (RLS-locked → only the service
  role; VIBE/anon can't see them), searched via `match_mase_document_chunks` RPC. NO
  `projects`/`documents` FK, NO `project_id` — a single MASE namespace. The old "MASE
  Knowledge" `projects` row was deleted so it no longer shows in VIBE.
- Backend: `mase_knowledge.py` (chunk + embed ada-002 + upload/list/get/delete);
  endpoints `GET/POST/DELETE /api/deal-engine/knowledge` + `GET /knowledge/{id}` (returns
  content reconstructed from chunks). Admin-gated at the proxy.
- `search_knowledge` routes the MASE namespace marker (`MASE_KNOWLEDGE_PROJECT_ID`, a
  routing constant only — NOT a real project) to `match_mase_document_chunks`. The
  todo-runner sends that project_id so its `search_knowledge` reads MASE knowledge.
- **Formats:** PDF, Word (.docx), **Excel (.xlsx/.xlsm)**, **PowerPoint (.pptx)**,
  CSV/TSV, Markdown, TXT, JSON/XML/YAML/log. Binary types extracted server-side in
  `_extract_text_from_file` (pypdf / python-docx / openpyxl / python-pptx). Legacy `.xls`
  → "save as .xlsx".
- Admin → Knowledge UI: **document-first list**, upload in a **modal**, per-doc **delete**,
  **click a doc to view its content** in a modal.

## 3. Admin → Agent Control

Tabs: Knowledge · Todo Runner · Deal Sweep · Execution · Access & Config. Admin-only
(gates on `isAdminView` = `realIsAdmin && !simEmail`, so simulating a rep/VP hides it).
- "**Tracked deals**" count (`GET /api/deal-engine/deals-count` → `count_active_records`).
- **Execution** shows TWO run feeds: Deal Sweep runs (sweep status + `trigger-logs`) and
  Todo Runner runs (`GET /api/deal-engine/todo-runner/runs`, status badges).
- The deal-filter bar is hidden on `/admin`.

## 4. Deals page performance

- The book loads **SLIM** (`?slim=1`): `hard` + `north_star_verdict` + `ai_fit_signal` +
  `pulse` only (~10-25x smaller). The full record is fetched per-deal on drawer open
  (`GET /opportunities/{id}`).
- The Deals table filters/sorts/paginates (**20/page**) client-side over the slim book;
  all 5 refinement filters work and the top search covers every deal. (A server-paginated
  endpoint `GET /opportunities?paged=1` exists, but the UI uses client-side because the
  AI-excitement filter is a computed tier the DB can't filter without a dedicated column.)

## 5. Reliability hardening

- **MCP per-tool timeout** (`MCP_TOOL_TIMEOUT_S` 300 API / 600 worker): a hung tool
  returns `{status:failed}` instead of pinning a run to the ~660s watchdog.
- **Pooled httpx + idempotency-safe retries** in `analysis_store.py` / `deal_engine_store.py`
  (`STORE_HTTP_RETRIES`); inserts never retry a maybe-landed error (no double-writes).
- **Graceful shutdown drain** (`server.py` shutdown event + `deploy.ps1 stopTimeout:120`):
  on deploy, in-flight runs are cancelled so each writes its own single terminal row —
  chats no longer hang on "Thinking…". (Hard SIGKILL/OOM still needs a startup reconciler
  — see `docs/enterprise-readiness.md` P1.1.)

## 7. Conventions & ops (read before changing things)

- **`deploy.ps1` ships the WORKING TREE, not GitHub.** A `git push` alone changes nothing
  in prod; a deploy ships whatever is on disk. Backend = blue/green ECS (slow drain now,
  due to stopTimeout 120). Frontend = Vercel auto-deploy on `main` push.
- **DDL** via the Supabase Management API (`SUPABASE_ACCESS_TOKEN` + `SUPABASE_PROJECT_REF`,
  `POST https://api.supabase.com/v1/projects/{ref}/database/query`). Use **curl** — Zscaler
  breaks Python's SSL verification. Never pass inline JSON to the AWS CLI from PowerShell.
- A new corpus `project_id` (VIBE-style) must exist in `projects` first (FK); MASE knowledge
  avoids this entirely via its own tables.
- Read `CHANGELOG.md` after every pull; append an entry when you change behaviour.
- Gotchas fixed this round: missing `projects` FK row (500), `mase_knowledge` importing a
  non-existent `config` module (500), backslash-in-f-string failing the <3.12 build image,
  `count_records` name collision.
