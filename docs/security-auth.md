# Security & Authentication

## Supabase RLS lockdown (added 2026-05-29)
Security audit found all 44 `public` tables had **RLS disabled** — so every table (incl. `avoma_event_reports`, `opportunity_observatory`, `opportunity_cache`, `chat_messages`, `documents`) was directly readable over the internet via Supabase's public REST API (`https://<ref>.supabase.co/rest/v1/...`) using only the publishable anon key, bypassing the server's Bearer auth entirely. The DB **already had 89 well-formed RLS policies** (e.g. "Users can view their own chats", "Project owners can manage memberships") — they were just dormant because RLS was off.

> ⚠️ **CURRENT STATE (2026-05-29, final): RLS is DISABLED on all 44 tables** at the user's explicit, informed request ("remove RLS lockdown" — confirmed they understand all user data + Salesforce/Avoma data is publicly readable over the anon REST API again). The 89 policies below remain in place but **dormant** (not enforced while RLS is off), so protection can be restored with a single `ALTER TABLE … ENABLE ROW LEVEL SECURITY` pass per table — no policy rebuild needed. The rest of this section documents the (now-reverted) lockdown for history.

**Fix (corrected 2026-05-29):** `ALTER TABLE … ENABLE ROW LEVEL SECURITY` on all 44 tables (applied via Management API DDL), activating the existing policies. **Table grants for `anon`/`authenticated` are LEFT AT SUPABASE DEFAULTS (`GRANT SELECT,INSERT,UPDATE,DELETE`)** — RLS, not the grant, is what scopes access per row. The server is unaffected either way because it connects with the **service-role key (BYPASSRLS)**.

⚠️ **Do NOT `REVOKE … FROM authenticated`.** An earlier pass revoked all grants on top of enabling RLS; that broke the live Vercel frontend (`zycus-deal.vercel.app`, ~134 Supabase-Auth users) — with no base grant, even valid logged-in users got permission-denied and saw empty Projects/Chats/404s. Grants were restored 2026-05-29; RLS stays ON.

Result: of the 44 tables, **28 have user-scoped policies** (projects, chats, chat_messages, workflows, profiles, user_daily_spend, …) → logged-in users see only their own rows; **16 have RLS on but no policies** (sensitive caches: `opportunity_cache`, `meeting_cache`, `avoma_event_reports`, `field_history_cache`, `opportunity_observatory`, `chat_usage`, `document_chunks`, …) → invisible to anon/authenticated over REST, readable only by the backend's service-role. Anon is still effectively blocked on sensitive tables (no anon-permissive policy). If a future frontend feature needs direct access to a no-policy table, add an explicit scoped RLS policy for it (don't disable RLS, don't revoke grants).

## Data/admin API authentication (added 2026-05-29)
Bearer-token auth on **all non-`/mcp` HTTP + WebSocket routes**, mirroring the `/mcp` `_MCPGateway`. Enforced in the outer ASGI wrapper `_AppWithMCP.__call__` (`server.py` ~line 7395) so it covers `/ws/chat` too (a FastAPI HTTP middleware would miss WebSockets). `/mcp` keeps its own `_MCPGateway` branch (handled first).
- **Token:** `API_AUTH_TOKEN` env, falling back to the same `DISPATCH_SECRET` `/mcp` uses — so one shared token by default and auth is active out of the box. If neither is set, the gate is fail-open (skips auth) and logs a loud `[API-AUTH] ⚠️ DISABLED` at startup; otherwise logs `[API-AUTH] ✅ ENABLED`. Comparison via `hmac.compare_digest` (constant-time).
- **Three credential sources** (`_api_auth_source`): `Authorization: Bearer <token>` (API/curl); `?key=<token>` query param (browser bootstrap — on a GET, the gate **302-redirects to the same path with `key` stripped** and sets an HttpOnly `api_auth` cookie, so the secret doesn't linger in the URL/history/proxy logs; non-GET sets the cookie inline); and the `api_auth` cookie (HttpOnly, `SameSite=Strict`, `Secure` when `X-Forwarded-Proto: https`, 30-day max-age). The cookie lets the HTML UIs (`/ask`, `/avoma/reports`) keep working with their plain same-origin `fetch()` calls.
- **Public allowlist (no auth):** exact `/api/health`, `/webhook` (AWS SNS can't send a Bearer), `/authorize`, `/oauth/token`, `/oauth/register`, `/oauth2callback`, `/auth/linkedin`, `/auth/linkedin/callback`; prefix `/.well-known/*`; plus all HTTP `OPTIONS` (CORS preflight). Everything else (incl. `/docs`, `/openapi.json`) requires the token.
- **User-requested PUBLIC exemptions (2026-05-29):** the user explicitly chose to leave these pre-existing (Feb/Mar) endpoints unauthenticated, treating them as externally-managed/not-owned: exact `/api/chat`, `/api/chat/async`, `/api/chat/structured/async`, `/api/config`, `/api/documents/upload`; prefix `/api/mcp/servers/` (the `{name}` POST/DELETE; the bare `GET /api/mcp/servers` list stays gated). ⚠️ Re-opens cost (chat) + control-plane (config) + add/remove-server + upload surfaces. Note `/api/chat/structured` (non-async) and `/api/chat/stop` are NOT exempted and remain gated. Remove these entries from `_API_AUTH_PUBLIC_EXACT` / `_API_AUTH_PUBLIC_PREFIX` to put them back behind the token.
- **On failure:** HTTP → `401` JSON + `WWW-Authenticate: Bearer`; WebSocket → close `1008`. Each rejection logs `[API-AUTH] 401 <method> <path>`.
- **CORS hardened alongside:** `allow_credentials` flipped `True`→**`False`** (server.py ~1899) — the invalid `allow_origins=["*"]` + credentials combo would otherwise reflect arbitrary origins for cookie-bearing requests. Bearer clients are cross-origin (no cookie needed); the UIs are same-origin (CORS N/A).
- **Verified 2026-05-29** (localhost curl): health=200 no-auth, protected=401 no-auth, Bearer=200, wrong-token=401, `?key=`→302+cookie (Location strips key), cookie reuse=200, `/ask?key=` redirect→cookie→`/api/ask`=200, `/mcp` still 401, live SNS `/webhook` still processes through enrichment.
- **Known follow-ups (NOT done here):** `/webhook` still lacks SNS signature verification + `SubscribeURL` host allowlist (SSRF/spoof risk — pre-existing TODO); no admin/role tiering (single token for read + write/DDL endpoints); `?key=` is a bootstrap convenience that still puts the secret in the request line once.
